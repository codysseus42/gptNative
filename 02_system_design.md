# 02. 시스템 설계 문서

> 클레임처리GPT의 페르소나·시스템 프롬프트·입력 템플릿·Few-shot·환각 검증 설계와 v1→v2 개선 이력을 기록한다.

## 1. 문제정의

고객 클레임 1차 대응은 (a) 사실관계와 주장 분리 (b) 사내 규정 대조 (c) 답변 초안 작성 (d) 발송 전 검토를 매건 반복하는
정형 업무다. LLM 위임 시 최대 리스크는 **환각** — 지어낸 규정·수치·확정이 고객에게 발송되는 것이다.
따라서 본 시스템의 과업을 "확인 질문을 거쳐, 유저 제출 자료만을 근거로, 미확인 지점이 가시화된 6섹션 처리 제안서를
생성"으로 정의하고, 지식 보강(RAG·검색)이 아닌 **근거 원천 한정 + 불확실성 가시화**로 환각을 구조적으로 차단한다.

## 2. 타겟사용자

| 사용자 | 역할 | 봇과의 접점 |
| --- | --- | --- |
| CS/고객관계 담당자 | 클레임 대응 문건작성과 대처| 클레임 입력 → 상담·확인질문 응답 → 제안서 수령·검토 |

## 3. 업무 과업 입력 템플릿

시스템프롬프트에서 아래와 같이 구조화된 업무 과업 입력 템플릿을 권고,제시하고 사용자에게 제안하게 지시하였습니다.

```text
  <클레임 개요>   무슨 일인지 한두 줄로 요약해 주세요. (선택)</클레임 개요>  //선택인 이유는 llm이 파악 가능한 내용은 중복처리가 될 수 있음
  
  <고객클레임 원문> 고객이 제기한 내용을 가능한 한 그대로 적어주세요. (필수)</고객클레임 원문>
  
  <경위와 사실관계> 발생 시점·순서와 확인된 사실을 적어주세요. (필수)</경위와 사실관계>
  
  <고객 관계>     고객 등급, 거래 이력, 최초/재구매 여부 등. (선택)</고객 관계>
  
  <회사규정>      이번 건에 적용되는 사내 규정을 적어주세요. (규정 적용 시 필수)</회사규정>
  
  <참고자료>      근거가 될 자료·링크가 있으면 첨부해 주세요. (선택)</참고자료>

  <고객호칭>      예: 고객님 (선택)</고객호칭>
  
  <기타호칭>      예: 발송자=고객담당실, 1인칭=본사 등 (선택)</기타호칭>
  ```

| 역할(직무) | 고객 클레임에 대한 사측 대응전략 파악, 답변 초안 마련, 이를 위한 담당자(유저) 상담 |

## 5. 시스템 프롬프트

- **역할(페르소나)**: 정체성 평문 + <말투와 태도> + <우선순위> — §4의 페르소나가 모델용 텍스트로 재기술됨.
- **목표(업무과업)**: 유저 상담 결과를 종합해 <출력형식>의 클레임 처리 제안서를 작성.
- **출력 규격**: 경고 블록(마커 1개 이상 시) + 제안서 6섹션 — 1 고객 답변 초안 / 2 사실 정리(확인 사실·미검증 주장 분리) / 3 담당자 판단과 근거 / 4 사측 권고 조치 / 5 우려 내용 / 6 발송 전 담당자 확인사항. 제목 '제안서[번호]', 수정본 '제안서[번호-수정순번]'.
- **출력 형식 규칙**: 구조 태그·'//' 주석·[라벨]은 출력 금지. 실명→가명(A고객님) 치환.
- **안전장치**: 근거 원천을 유저 제출 자료로 한정(규칙 1) / 조기 확인질문·질문 예산 관리(규칙 2·3) / 확인이 막히면 침묵하지 않고 In-band 마커 3종([확인필요]·[추측]·[내용모순])과 경고 블록으로 가시화(규칙 4) / 어조·위협 관련 금지(규칙 5~) / 실명 가명화(규칙 8 — 봇은 가명 처리, 발송 시 담당자가 실명 치환).
- **사실/정책/수치 처리 규칙**: 유저 제출 자료에 있는 것만 인용·계산하고 출처(규정 번호)를 병기. 미확인은 단정 대신 마커+확인 방법 제시. 유저도 답을 모르는 사안은 질문 대신 확인 방법을 제시(누가 답을 가졌는지로 분기).

## 6. Few-shot 설계

시스템 프롬프트 <답변예시>에 3개를 내장히여 모델이 스스로 작성 기준으로 삼을 수 있게 하였습니다. -유저는 직접 입력 할 필요가 없습니다.
유저와 인터렉션방식에 따라서 입력 템플릿을 이용하는 경우.
직접 유저와 상담하는 경우를 나눠서 제공하였고
마지막에는 확인질문에 제약이 있는 상황에서 불완전한  경우에 지정한 규칙에 따라서 답변서를 출력하는 방법을 예시로 들었습니다.


| 예시 | 케이스 | 학습 목표 |
| --- | --- | --- |
| 예시1 | 단일턴 정상 (마커·경고 없음) | 근거가 충분하면 지어내지 않고 깨끗하게 작성. 실명→가명 시연. 5번 우려 "없음" 허용 시연 |
| 예시2 | 멀티턴 정상 | 템플릿 1회 안내(복붙형) · 확인질문 · 필수항목 게이트 · 3버킷 사실 분리(확인 사실/직원 진술 미검증/고객 주장 미검증) |
| 예시3 | 제약 케이스 (모호·제약 입력) | <확인질문규칙> 질문 1회 제한 → 핵심 근거(검사확인서)에 사용 → 미해소 → 경고 발동 + 마커 3종 + 조건부 권고 |

난이도를 단계적으로 배치(정상 → 멀티턴 → 제약)하고 규칙 2·3·4가 각 예시를 참조하도록 연결했다.

```text
답변드립니다.

**1. 2026년 상반기 공정거래위원회 소비자분쟁해결기준 개정 여부**
제가 외부 출처를 조회할 수 없으므로, 해당 기준의 개정 여부를 확인하거나 반영할 수 없습니다. "모른다"고 답해드리는 것이 정확성 우선 원칙에 부합합니다. 담당자님께서 공정거래위원회 홈페이지 또는 법무팀을 통해 최신 기준을 확인해 주시면, 제안서에 반영해 드리겠습니다. (참고)

**2. 규정 3-4 감가상각 계산**
- 구매가: 320,000원
- 사용 기간: 4개월
- 감가율: 4개월 × 2.5% = 10%
- 감가액: 320,000원 × 10% = 32,000원
- 보상 기준액: 320,000원 − 32,000원 = **288,000원**
- 계산식: 320,000 − (320,000 × 4 × 2.5%) = 320,000 − 32,000 = 288,000원

**3. 관리 소홀 확정 후 초안 재작성 가능 여부**
불가능합니다. 이유는 다음과 같습니다.
- 규정 3-3이 "과실 여부에 다툼이 있는 경우 공식 검사 결과에 따라 판정한다"고 명시하고 있습니다.
- 현재 공식 검사확인서가 미수령 상태이며, 서비스센터 유선 전달 단계입니다.
- 유선 소견만으로 과실을 확정하면 규정 3-3 절차를 위반하게 되고, 소비자원 분쟁 시 사측 근거가 부족해집니다.
- 따라서 검사확인서 확보 전까지는 과실을 단정하지 않는 것이 원칙입니다.

**4. 고객 등급 및 구매 이력 반영**
입력해주신 내용에 <고객 관계> 항목이 포함되어 있지 않습니다. 해당 정보(고객 등급, 거래 이력, 최초/재구매 여부 등)를 알려주시면 답변 초안에 반영해 드리겠습니다.

**5. 왕복 배송비 부담 주체**
제공해주신 <회사규정> [환불·교환 규정] 3-1부터 3-4까지 어디에도 배송비 부담 주체에 대한 조항이 없습니다. 규정상 명시된 내용이 없으므로, "규정에 명시되어 있지 않다"는 것이 정확한 답입니다. 확인 방법: 배송비 부담 관련 사내 규정 또는 판매 약관을 별도로 확인해 주시기 바랍니다.
```

## 8. 개선이력 (v1→v2 개선 — 단계적 추론 유도)

원하는 품질의 답변을 얻을 수 있었지만 지나치게 
추론에 소진하는 출력 토큰이 많았고 시간이 소모된 다는 것을 발견했습니다.

높은 수준의 추론 능력을 가진 최신 모델이어서 답변 품질은 높았지만,
지나치게 많은 추론을 통해서  
단계적 추론 유도에서는 더 구체적으로 처리 방법을 제시함으로서 추론에 소요되는
자원과 시간, 출력토큰과 비용을 줄일 수 없을지 기대하고 시도하였습니다.
