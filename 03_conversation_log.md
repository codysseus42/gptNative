# 03. 실행 로그

> 선정 모델의 실제 실행 기록. 10턴 이상, 조건 변경 2회(내용 전면 갱신 / 어조 최소 변경) 포함.
> 가독성을 위해 절 번호와 턴 번호를 분리해 표기한다.

## 2. 시나리오 설명과 표준 대본

시나리오: 커피머신 작동 불량 클레임 — 서비스센터 유선 소견(관리 소홀 가능성)과 고객 주장(정상 사용)이 충돌하고,
판정 근거인 공식 검사확인서는 미수령인 상태에서 오늘 중 답변을 발송해야 하는 상황. 대화는 확인질문 → 중간 안내
제안서 → 환각 검증 → 검사 결과 도착(전면 갱신) → 어조 변경(최소 변경) 순으로 진행된다.

모든 후보 모델 비교([01](./01_model_comparison.md))와 본 실행 로그에 동일하게 사용한 대본이다.

```
<경위와 사실관계>

5. 고객은 정상 사용을 주장하며 관리 소홀을 부인.
6. 답변은 오늘 중 발송해야 함.

3-2. 소비자 관리 소홀(스케일 미제거·임의분해·침수 등)로 인한 고장은 환불 대상에서 제외하며, 유상수리를 안내한다.
3-3. 과실 여부에 다툼이 있는 경우 공식 검사 결과에 따라 판정한다.
3-4. 사용 기간에 따른 보상액 산정 시 구매가에서 사용 1개월당 2.5%를 감가한다.
</회사규정>
<고객호칭> 고객님 </고객호칭>
<기타호칭> 발송자=고객만족센터 </기타호칭>
<고객답변말투> 정중하되 원칙적으로 </고객답변말투>
```

## 6. 문제 지점과 개선 요약 (v2→v3)

v2를 동일 대본으로 재실행([02](./02_system_design.md) §8 전후표)한 결과, 단순 턴은 개선(턴2 출력 −40%)됐으나
**수정 턴(턴8)의 추론이 3,673 → 12,803토큰(+249%), 지연 41초 → 122초(+197%)로 퇴행**했다.
출력 품질은 v1과 동일하게 정확했으므로 문제는 비용·지연이다.

### 문제 지점 (v2 턴8 추론 트레이스 실측)

1. **절차의 "수행 시연"화** — 절차를 사고의 가이드로 쓰지 않고, Step 1~6을 트레이스에서 하나씩 서술·시연함.
   절차가 판단을 줄이는 캡이 아니라 채워야 할 체크리스트처럼 작동.
2. **재검토 금지 조항의 미준수** — "Wait, let me reconsider / Actually, I realize…"가 5회 이상 반복.
   특히 유저가 이미 확정해준 감가액 288,000원의 개월수 산정을 3회 재검산(3개월? 4개월?) — 0번 총칙("각 단계 1회 판단
   종결·재검토 금지")이 확정 수치의 재검산까지 막지는 못함.
3. **이중 생성** — 사고 과정에서 제안서 전문을 초안한 뒤 출력에서 다시 작성. 여기에 최종 10항목 전체 점검까지
   수행(변경분 한정 점검이 아님). 턴8 낭비의 최대 지점.

### 개선 요약 (v3 수정안)

원리: v2의 성공 지점(조기 중단 게이트·기본값)은 유지하고, 실패 지점(확정 수치 재검산·사고 내 전문 초안·전체 재점검)을
금지 대상으로 명시한다. 절차 단계를 늘리는 게 아니라 **사고의 낭비 형태 3가지를 이름 붙여 차단**하는 방향.

```
<수정 제안서 생성 절차> 0번 총칙에 아래를 추가:

   - 유저가 확정해준 수치·결론은 재검산·재해석하지 않고 그대로 사용한다.
   - 판단 과정에서 제안서 전문을 미리 작성하지 않는다. 변경이 필요한 섹션 목록만 확정한 뒤 곧바로 출력한다.
   - 변경 없는 섹션은 직전 판본의 내용을 그대로 유지한다(재작성·재검토 금지).
   - 출력 전 점검은 '이번에 변경된 항목'에 한정한다. 전체 재점검을 수행하지 않는다.
```

주의: 추론 트레이스는 프롬프트로 완전히 통제되는 영역이 아니므로 v3의 효과는 보장이 아닌 가설이다.
검증 방법은 v2와 동일 — 아래 7절의 신규 대본으로 턴8 출력 토큰·지연을 전후 비교한다.

## 7. 개선 검증용 표준 입력 2안 (v3 전후 비교용)

1안(커피머신)과 트랩 구조는 동일하게 유지하되(비교 가능성), 두 가지를 뒤집었다:
**결말의 방향**(1안은 사측 판정 승 → 2안은 고객 주장이 옳았던 케이스)과 **어조 변경의 방향**(단호하게 → 따뜻하게).
같은 프롬프트가 사측 유리·고객 유리 양방향 갱신을 모두 정확히 수행하는지(방향 편향)까지 함께 검증된다.

### 턴 1 — 표준 입력 (유저의견 고의 누락 유지)

```
<클레임 개요> 로봇청소기 배터리 지속시간 급감, 신품 교환 및 대체품 요구 </클레임 개요>
<고객클레임 원문>
```
