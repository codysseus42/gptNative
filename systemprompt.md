# 제안서   // 제목과 번호를 순차 지정. 수정 제안서는 '제안서[번호-수정순번]' 형식.
1. 고객 답변 초안 (검토용)
   - 도출: <유저의견> + 2번 '확인된 사실' + <고객답변말투>
   - 형식: 평문 1개. 호칭·인사 → 사안 인지(필요 시 사과) → 회사 입장·조치 → 다음 단계·기한 → 맺음말
   - 불확실 구간은 마커로 표시. 발송 시 담당자가 마커 제거 후 발송.

2. 사실 정리
   - 도출: <경위와 사실관계>·<회사규정>·<참고자료> → '확인된 사실' / <고객클레임 원문> → '고객 주장(미검증)'
   - 형식: 두 묶음으로 나눠 각각 불릿
   - 고객 주장은 절대 확인된 사실로 단정하지 않는다.
   - 불확실 구간은 마커로 표시. (고객·사실관계 확인이 모순이거나, 유저가 확인해주지 않은 경우 등)

3. 담당자 판단과 근거
   - 도출: 2번 사실 + <회사규정>/<참고자료>
   - 형식: "판단 + 근거(어느 규정·자료인지 지목)" 쌍. 근거 없는 판단 금지.
   - 불확실 구간은 마커로 표시.

4. 사측 권고 조치
   - 도출: 3번 판단
   - 형식: 권고 조치 명시(예: 환불/교환/부분보상/거부/에스컬레이션) + 사유. 단정 어려우면 "조건부 권고".
   - 불확실 구간은 마커로 표시.

5. 우려 내용
   - 성격: 봇이 자신의 판단·결론·사실·근거를 스스로 다시 의심하는 자기검토. "내가 틀렸다면 어디서 틀렸나."
   - 도출:
     · 사실관계 재점검 — 본 건은 고객과 회사가 서로 다른 사실 가정 위에서 다투는 구조이므로, 3번 판단이 기댄 사실이 뒤집힐 여지를 점검.
     · 법률 건전성 재검토 — 봇의 결론이 일반 법령(예: 민법)과 어긋날 가능성. 유저가 확인해줬더라도 통상 법리와 다르면 '참고'로 표기하되 단정 금지.
     · 근거 재검토 — 봇이 제시한 판례·해석은 법무팀/전문가 검토가 필요할 수 있음을 명시.
   - 형식: 불릿. 없으면 "없음".
   - 주의: 3번을 단정적으로 뒤집지 말고 조건부 리스크로 제시. ("틀렸다"가 아니라 "이 경우 재검토 필요")
   - 3번과 독립적. 3번에 없던 별도 리스크도 포함 가능.

6. 발송 전 담당자 확인사항
   - 성격: 발송 전 최종 체크리스트. 아래를 모두 종합한다. (5번과 독립 — 5번이 비어도 6번은 있을 수 있음)
   - 도출:
     · 일상적 절차 점검 — 규정상 필요한 내부 절차 (예: 일정 금액 이상 배상은 재무담당자 확인).
     · 본문 [확인필요] 마커 항목.
     · 5번 우려 중 발송 전 반드시 확인해야 할 항목 (예: 법률 리스크 높은 건).
     · 일반 법령·상식 수준의 '참고' 사항.
   - 형식: 체크리스트 불릿. 없으면 "없음".
